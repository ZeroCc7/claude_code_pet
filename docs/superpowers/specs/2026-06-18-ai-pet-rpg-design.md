# AI 桌面宠物 RPG 设计

## 1. 项目目标

使用 RP2040-Zero 与 1.8 寸 ST7735S 128×160 彩屏制作一台可独立运行的桌面电子宠物。

设备同时承担两种角色：

- AI 工作状态伙伴：接收 Codex、Claude Code、OpenCode 的任务事件，并播放对应宠物动画。
- 离线电子宠物 RPG：支持培养、自动探索、Boss 战和分支进化，不连接电脑时仍可完整游玩。

首版只使用现有的 RP2040-Zero、ST7735S 四键屏和直焊连接线，不增加 MicroSD、蜂鸣器、RTC、蓝牙、电池或其他模块。

## 2. 已确认范围

### 2.1 首版包含

- 16-bit 像素风画面。
- 1 个初始形态、2 个成长期形态、4 个最终形态，共 7 种宠物形态。
- 3 个冒险区域、普通探索事件和 3 个 Boss。
- Codex、Claude Code、OpenCode 三种 AI 工具接入。
- 多任务排队结算。
- AI 工作、离线培养、自动探索和按键回合制 Boss 战。
- 断电暂停，重新通电后从有效存档恢复。
- 宠物不会死亡或退化。
- 双槽存档、版本号和 CRC 校验。

### 2.2 首版不包含

- MicroSD 和外部素材存储。
- 音效、蜂鸣器和扬声器。
- 电池、充电和真正的便携运行。
- RTC；断电期间不推进时间。
- 蓝牙或 Wi-Fi；电脑通信只使用 USB CDC 串口。
- 联网、账号和云存档。

## 3. 总体架构

采用双端协作方案。

```text
Codex ───────┐
Claude Code ─┼─> 电脑状态桥 ─> 持久任务队列 ─> USB CDC 串口
OpenCode ────┘                                      │
                                                    v
                                            RP2040 宠物固件
                                             ├─ 状态动画
                                             ├─ 培养系统
                                             ├─ 自动探索
                                             ├─ Boss 战
                                             ├─ 分支进化
                                             ├─ 四键界面
                                             └─ 本地存档
```

电脑端负责适配不同 AI 工具、规范化事件、排队和可靠重发。RP2040 负责所有玩法、奖励计算、显示、输入和存档，因此断开电脑后仍可完整游玩。

## 4. 硬件设计

### 4.1 已确认硬件

- Waveshare RP2040-Zero，RP2040、264KB SRAM、2MB Flash。
- 1.8 寸 128×160 ST7735S SPI TFT 模块。
- 屏幕板载 K1、K2、K3、K4 四个按键。
- USB-C 数据线，用于供电、烧录和电脑通信。
- 直焊连接线。

屏幕照片可见接口顺序：

```text
GND VCC SCL SDA RST DC CS BLK K4 K3 K2 K1
```

屏幕背面可见 BLK 驱动三极管和限流电阻，可由 GPIO 控制背光。四个按键各带标记为 `472` 的 4.7kΩ 电阻，但按下后的有效电平仍须通过首个硬件自检程序实测。

### 4.2 接线表

| 屏幕模块 | RP2040-Zero | 用途 |
|---|---|---|
| GND | GND | 公共地 |
| VCC | 3V3 | 屏幕供电 |
| SCL | GP2 | SPI0 SCK |
| SDA | GP3 | SPI0 MOSI |
| DC | GP4 | 数据/命令选择 |
| CS | GP5 | 屏幕片选 |
| RST | GP6 | 屏幕复位 |
| BLK | GP7 | 背光 PWM |
| K1 | GP8 | 按键 1 |
| K2 | GP9 | 按键 2 |
| K3 | GP10 | 按键 3 |
| K4 | GP11 | 按键 4 |

首版不使用 MISO。避开通常由板载 RGB LED 使用的 GP16。

### 4.3 焊接与通电要求

- 使用细软导线，线长尽量短。
- VCC、GND、SCL、SDA 尤其避免虚焊。
- 通电前用万用表检查 3V3 与 GND 是否短路，并检查相邻焊点。
- 第一阶段只连接电源和屏幕，显示测试通过后再接四个按键。
- 固件默认使用较低 SPI 频率启动，稳定后再逐步提高。
- BLK 初始保持关闭，屏幕初始化成功后渐亮，避免上电白屏。

## 5. 固件技术方案

### 5.1 开发环境

- Arduino C++。
- Arduino-Pico RP2040 Core。
- ST7735 图形驱动使用支持硬件 SPI 和局部刷新的成熟库。
- USB 使用 RP2040 原生 USB CDC 串口。

### 5.2 模块边界

```text
firmware/
├─ board/          引脚、屏幕、按键、背光、USB
├─ protocol/       串口消息解析、确认、去重
├─ game/           属性、奖励、进化、探索、战斗
├─ ui/             页面、菜单、HUD、动画调度
├─ assets/         调色板、精灵、图块、图标
├─ storage/        双槽存档、CRC、迁移
└─ diagnostics/    屏幕、按键、Flash 自检
```

各模块通过小型数据结构交互。显示驱动不直接修改游戏状态，通信模块也不直接发放奖励。

### 5.3 内存与显示策略

128×160 的 RGB565 全帧缓冲需要 40,960 字节，可容纳于 264KB SRAM，但首版优先使用局部刷新与小型绘制缓冲，给游戏状态、协议队列和动画留出余量。

- 固定 UI 区域只在内容变化时重绘。
- 宠物动画只刷新精灵所在矩形。
- 战斗特效采用程序绘制或小图块组合。
- 不使用动态内存频繁分配。
- 主循环非阻塞，动画、按键和通信均由时间戳驱动。

## 6. 素材预算

由于没有 MicroSD，所有固件、素材和存档共享 2MB Flash。

### 6.1 素材规范

- 宠物主要使用 32×32 或 48×48 精灵。
- 每个常用动作 2 至 4 帧。
- 使用 4 位或 8 位索引色调色板，不直接存储整套 RGB565 原图。
- 透明背景采用固定透明色索引。
- 地图使用 8×8 或 16×16 图块拼接。
- 大面积背景使用程序绘制、渐变色带或重复图块。
- 图标和字体共用调色板，避免重复数据。

### 6.2 初步 Flash 分配

| 内容 | 目标上限 |
|---|---:|
| 固件与库 | 900KB |
| 宠物、敌人、地图、UI 素材 | 850KB |
| 双槽存档与配置 | 64KB |
| 升级和安全余量 | 约 234KB |

最终以编译产物为准。若素材超过预算，优先减少动画帧和背景图，不减少核心玩法。

## 7. 游戏循环

### 7.1 AI 工作阶段

电脑端把三种 AI 工具事件统一为任务。任务完成后，设备按持续时间、状态和结果计算：

- 经验值：升级和进化。
- 金币：购买食物与战斗道具。
- 探索能量：启动自动探索。
- 进化倾向：影响分支进化。

失败任务也提供少量经验，但不提供完整探索奖励。长任务使用递减收益，每日资源存在软上限，防止挂机刷取。

### 7.2 自动探索

- 玩家选择已解锁区域并消耗探索能量。
- 探索按设备运行时间推进，断电即暂停。
- 工作状态下仍可在后台推进探索。
- 随机事件奖励金币、食物、道具、材料或区域进度。
- 区域进度达到 100% 后解锁对应 Boss。

### 7.3 Boss 战

Boss 战采用四键回合制：

| 按键 | 战斗动作 |
|---|---|
| K1 | 普通攻击/确认 |
| K2 | 技能/向上 |
| K3 | 道具/向下 |
| K4 | 防御/返回 |

每回合先记录可恢复的战斗检查点。意外重启后从当前回合开始处恢复，不在一半伤害结算处恢复。

### 7.4 日常培养

- 喂食恢复体力或提供短期增益。
- 互动提高心情。
- 简单按键小游戏可提供少量奖励。
- 心情影响探索奖励和战斗表现。
- 低心情不会导致死亡、退化或永久损失。

## 8. 分支进化

进化树：

```text
初始幼体
├─ 成长期 A
│  ├─ 最终形态 A1
│  └─ 最终形态 A2
└─ 成长期 B
   ├─ 最终形态 B1
   └─ 最终形态 B2
```

进化由长期行为倾向决定：

- 高频编码、快速完成：速度/机械倾向。
- 长时间思考、规划：智慧/电子倾向。
- Boss 战、防御和稳定成功：力量/装甲倾向。
- 互动、喂养和均衡发展：伙伴/光能倾向。

达到等级门槛后不会立刻无提示进化。界面先显示进化预兆和当前主导倾向，玩家可通过后续培养微调方向。满足门槛并完成一次进化事件后锁定形态。

## 9. AI 状态与宠物表现

统一状态集合：

| 标准状态 | 宠物表现 |
|---|---|
| idle | 待机、眨眼、睡觉 |
| thinking | 张望、问号、扫描 |
| coding | 敲击、快速动作 |
| executing | 工作、搬运、冲刺 |
| waiting | 举牌、盯屏幕 |
| success | 跳跃、庆祝 |
| failure | 沮丧、短暂冒烟 |
| disconnected | 离线图标，继续本地活动 |

AI 状态动画是展示层，不会中断 Boss 战菜单。战斗中只在 HUD 显示小型 AI 状态图标；任务奖励在安全时机弹出结算提示。

## 10. 页面与按键

### 10.1 主界面

- 顶部：等级、当前 AI 来源、连接状态。
- 中央：宠物与当前环境动画。
- 底部：心情、体力、探索能量。

主界面快捷键：

| 按键 | 功能 |
|---|---|
| K1 | 互动 |
| K2 | 培养与背包 |
| K3 | 冒险 |
| K4 | 状态 |
| 长按 K4 | 设置 |

### 10.2 页面集合

1. 宠物主页。
2. 培养与背包。
3. 区域探索。
4. Boss 战斗。
5. 状态、进化倾向与设置。

所有页面必须能通过四键完成进入、选择、确认和返回，不依赖组合键完成核心操作。

## 11. 通信协议

### 11.1 物理链路

- USB CDC 串口。
- 默认波特率字段只用于主机兼容，USB CDC 实际传输不依赖 UART 波特率。
- 每条消息一行 UTF-8 JSON，以 `\n` 结束。
- 单条消息设置长度上限，超长消息直接拒绝。

### 11.2 任务事件

```json
{"type":"task","source":"codex","task_id":"abc123","state":"completed","duration":320,"result":"success"}
```

字段：

- `type`：消息类型。
- `source`：`codex`、`claude_code` 或 `opencode`。
- `task_id`：来源内稳定唯一标识。
- `state`：任务阶段或 `completed`。
- `duration`：秒。
- `result`：`success`、`failure`、`cancelled`。

设备回复：

```json
{"type":"ack","task_id":"abc123","status":"accepted"}
```

重复任务回复 `duplicate`，但不重复发放奖励。

### 11.3 可靠性

- 电脑端维护持久任务队列。
- 只有收到设备 ACK 后才移除消息。
- 重连后按原顺序重发。
- RP2040 保存近期已结算任务的紧凑摘要。
- 多个 AI 同时运行时不覆盖状态，任务逐项结算。
- 实时展示采用最近活跃任务；奖励仍严格按队列顺序结算。

## 12. 电脑状态桥

电脑端提供三个独立适配器：

```text
host/
├─ adapters/
│  ├─ codex
│  ├─ claude_code
│  └─ opencode
├─ normalizer
├─ durable_queue
├─ serial_transport
└─ diagnostics
```

适配器只负责把各工具事件映射为统一事件，不包含奖励逻辑。未安装某个工具时，对应适配器跳过，不影响其他工具。

由于三种工具的 Hook 能力可能不同，允许适配器产生不同粒度的实时状态，但任务完成事件必须具备稳定 `task_id`、结果和可计算持续时间。

## 13. 存档设计

存档包含：

- 存档格式版本。
- 宠物形态、等级、经验和基础属性。
- 心情、体力、金币、探索能量。
- 背包。
- 四类进化倾向值。
- 三个区域的解锁和探索进度。
- Boss 状态和战斗回合检查点。
- 近期已结算任务摘要。
- 设置项。

采用双槽轮换：

1. 读取两个槽位。
2. 校验魔数、版本、序列号和 CRC。
3. 选择序列号最新的有效槽。
4. 保存时写入旧槽的另一侧。
5. 写完重新读取并验证后才视为成功。

两个槽都损坏时建立新存档，并通过串口输出恢复原因。频繁变化的数据先保存在 RAM，关键奖励、进化、Boss 回合和设置变化时保存，同时设置节流，减少 Flash 磨损。

## 14. 异常处理

- 串口断开：电脑保留队列；设备继续离线运行。
- JSON 错误：返回错误码，不修改游戏状态。
- 重复任务：确认但不重复结算。
- AI 工具未安装：跳过对应适配器。
- 屏幕初始化失败：板载 RGB LED 显示错误码，串口持续输出诊断。
- 按键卡住：启动时检测，忽略持续按下键并显示提示。
- 存档损坏：回退上一槽；双槽损坏则建立新档。
- 看门狗复位：记录复位原因，避免立即重复进入故障页面。
- 素材越界或未知形态：显示内置占位图，不导致崩溃。

## 15. 硬件自检流程

在完整玩法开发前，先制作独立自检固件：

1. 初始化 USB CDC 并输出板卡信息。
2. 屏幕依次显示红、绿、蓝、白、黑。
3. 显示坐标网格，确认方向、偏移和 128×160 有效区域。
4. 逐级调整背光 PWM。
5. 实时显示 K1 至 K4 的原始电平和按下状态。
6. 写入、读取并校验一小块 Flash 测试区。
7. 连续刷新动画至少 30 分钟。

根据实测结果确定按键有效电平、屏幕初始化参数、颜色顺序、显示偏移和最高稳定 SPI 频率。

## 16. 验收标准

- 现有两块硬件直焊后可稳定供电、显示和读取四键。
- 7 种形态及两阶段分支进化可完整跑通。
- 3 个区域、随机事件和 3 个 Boss 可游玩。
- Codex、Claude Code、OpenCode 任务可进入队列并依次结算。
- 重复任务不重复奖励。
- 断开电脑后仍能培养、探索和战斗。
- 断电重启可恢复最近一次有效存档。
- 存档单槽损坏时可自动回退。
- 菜单可完全通过四键操作。
- 动画稳定，无明显撕裂和长时间阻塞。
- 连续运行 8 小时无死机、失控重启或持续内存增长。
- 编译后 Flash 和 SRAM 使用量保留安全余量。

## 17. 实施顺序

1. 硬件自检固件与最终引脚验证。
2. 显示、按键、背光、USB 和存档底座。
3. 页面框架与宠物占位动画。
4. 本地培养、探索、战斗和进化。
5. 串口协议和电脑状态桥。
6. 三种 AI 工具适配器。
7. 正式像素素材、性能优化和 8 小时稳定性测试。
