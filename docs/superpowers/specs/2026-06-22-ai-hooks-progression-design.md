# AI Hook、持续成长与进化设计

## 1. 目标

缩短灵卵阶段，补齐宠物在成长期以后缺少持续经验来源的问题，并让 Codex、Claude Code、OpenCode 的完整任务成功事件成为修炼奖励来源。

本轮完成：

- 历练持续获得经验。
- 已击败首领可以重复挑战，奖励逐次减半。
- LV3、LV12 自动进化，并明确提示玩家。
- 接入三种 AI 工具能够提供的多类 Hook。
- AI 工作期间显示独立修炼状态页。
- 只有完整任务成功结束才结算经验和灵石。

本轮不制作正式动画素材。动画状态、触发时机和表现要求先固定，后续单独生成素材并实现。

## 2. 成长与进化

### 2.1 等级

- 每累计 20 点经验提升一级。
- 最高等级保持 30。
- LV3：初始形态自动进入成长期，只需累计 40 点经验。
- LV12：成长期自动进入最终形态。
- 分支继续由现有四类倾向决定，不增加人工选择页面。

### 2.2 历练经验

已启动的历练每分钟结算一次：

| 区域 | 每次进度 | 每分钟经验 |
|---|---:|---:|
| 区域一 | 现有随机进度 | 1 |
| 区域二 | 现有随机进度 | 2 |
| 区域三 | 现有随机进度 | 3 |

历练仍消耗灵力。灵力耗尽或区域进度达到 100% 时自动停止。

### 2.3 重复首领

- 首次击败仍负责解锁下一区域。
- 已击败的首领可再次进入战斗。
- 每个区域分别记录累计胜利次数。
- 第一次及后续胜利的经验和灵石按 `基础奖励 / 2^重复次数` 计算，整数向下取整，最低各为 1。
- 基础奖励沿用现有值：
  - 经验：`20 + 区域编号 × 10`
  - 灵石：`20 + 区域编号 × 15`
- 首次胜利为 100%，第二次 50%，第三次 25%，之后继续减半。

首领胜利次数进入存档。存档版本升级，并兼容现有旧存档；旧存档迁移后，已击败区域按首次胜利完成处理。

### 2.4 进化反馈

任何奖励结算前记录旧形态，结算后比较新形态：

- 形态未变化：显示普通奖励提示。
- 形态变化：立即请求存档，进入非阻塞进化提示。
- 首版提示包括新形态名称、等级和短暂闪光，不阻塞按键扫描与存档。
- 正式进化动画留到后续美术阶段。

## 3. AI 任务奖励

### 3.1 成功条件

仅完整任务成功结束时奖励。思考、工具调用、文件修改、等待用户等中间事件只更新屏幕状态。

失败、取消、权限拒绝不发成功奖励。

### 3.2 奖励公式

任务耗时按分钟计算：

```text
分钟数 = clamp(floor(持续秒数 / 60), 1, 60)
经验 = 分钟数 × 2
灵石 = 分钟数
```

沿用现有 `applyTask()` 奖励逻辑。成功任务可以恢复少量灵力，但不能超过 20。

### 3.3 防重复

- 每个完成事件包含 `source` 和稳定的 `task_id`。
- 固件保存近期 16 个任务摘要。
- 相同 `source + task_id` 再次到达时返回 `duplicate`，不重复结算。
- 近期任务摘要进入存档，断电重启后仍能防止重复奖励。

## 4. 串口协议

每条消息为一行 UTF-8 JSON，以换行结束，最大 384 字节。

### 4.1 状态事件

```json
{"type":"status","source":"codex","task_id":"abc123","state":"thinking","timestamp":1782000000}
```

支持状态：

| 状态 | 修仙文案 | 用途 |
|---|---|---|
| `submitted` | 接引任务 | 用户提交请求 |
| `thinking` | 推演中 | 模型思考或生成 |
| `tool` | 施法中 | 调用一般工具 |
| `editing` | 炼器中 | 写入或修改文件 |
| `waiting` | 等待指令 | 等待用户确认或输入 |
| `blocked` | 修炼受阻 | 权限拒绝、工具失败 |
| `idle` | 修炼暂歇 | 会话结束或空闲 |

状态事件不修改经验和资源。

### 4.2 完成事件

```json
{"type":"task","source":"codex","task_id":"abc123","state":"completed","duration":320,"result":"success"}
```

固件回复：

```json
{"type":"ack","task_id":"abc123","status":"accepted"}
```

重复任务回复：

```json
{"type":"ack","task_id":"abc123","status":"duplicate"}
```

无效 JSON、未知来源、字段缺失或超长消息返回错误 ACK，不修改游戏状态。

## 5. Hook 接入

采用各工具 Hook 直接调用统一 PowerShell 发送脚本，不运行常驻服务。

### 5.1 公共发送层

- 安装位置：`%USERPROFILE%\.ai-pet-hooks\`
- 公共脚本负责：
  - 构造统一事件。
  - 计算或传递任务 ID、开始时间和持续时间。
  - 自动发现 RP2040 串口，允许配置固定端口。
  - 使用 Windows 命名互斥锁，避免多个 Hook 同时抢占 COM 口。
  - 短暂等待设备 ACK。
  - 写入本地日志。
- 串口不可用时 Hook 快速退出，不阻塞 AI 工具正常工作。

### 5.2 Codex

安装脚本先检测当前 Codex 版本实际支持的生命周期 Hook，并尽量映射：

- 用户提交：`submitted`
- 模型思考：`thinking`
- 工具调用：`tool`
- 文件修改类工具：`editing`
- 等待用户：`waiting`
- 工具失败或拒绝：`blocked`
- 成功结束：发送 `completed + success`
- 取消或失败：发送失败状态，不奖励
- 会话结束：`idle`

不支持的事件不伪造；若当前版本只能提供结束通知，则先接入可靠的完成事件，其余状态等待工具升级后自动启用。保留现有 Codex 桌面通知链，不覆盖 CursorLight 的 `notify` 配置。安装脚本只增量合并检测到的 Hook。

### 5.3 Claude Code

使用官方 Hook 事件映射：

- `UserPromptSubmit` → `submitted`
- `PreToolUse` → `tool`，文件写入工具映射为 `editing`
- `PostToolUseFailure`、权限拒绝 → `blocked`
- 通知或询问用户 → `waiting`
- `Stop` 且确认成功 → 完成奖励
- `SessionEnd` → `idle`

安装脚本备份并合并 `~/.claude/settings.json`，不覆盖用户已有 Hook。

### 5.4 OpenCode

使用插件事件映射其可用生命周期事件：

- 消息提交 → `submitted`
- 工具执行 → `tool` 或 `editing`
- 权限请求 → `waiting`
- 工具错误 → `blocked`
- 会话进入 idle 且本轮成功 → 完成奖励
- 会话结束 → `idle`

若本机尚未安装 OpenCode，只生成配置模板和安装说明，不制造不存在的配置目录。

## 6. 修炼状态页

收到非 `idle` 状态后切换到独立修炼页。页面包括：

- 当前来源：Codex、Claude Code 或 OpenCode。
- 当前修炼状态和对应中文文案。
- 宠物当前形态。
- 本次任务已持续时间。
- 简单的程序绘制动效，占用较少 Flash。

收到成功完成事件：

- 显示“修炼完成”。
- 显示本次经验和灵石增量。
- 若发生进化，优先进化提示。
- 提示结束后返回首页。

收到 `idle`、失败或超时：

- 短暂显示对应结果。
- 不发成功奖励。
- 自动返回首页。

设备长时间没有收到后续 Hook 时，状态页最多保持 10 分钟，之后自动回首页，避免永久卡在“推演中”。

## 7. 后续动画清单

后续美术阶段为以下状态分别设计动画：

| 状态 | 动画方向 |
|---|---|
| 接引任务 | 符箓展开、灵光汇入 |
| 推演中 | 八卦或星盘旋转、宠物闭目推演 |
| 施法中 | 法阵脉冲、灵气射出 |
| 炼器中 | 金色火花、器纹逐步点亮 |
| 等待指令 | 悬浮问符、呼吸光 |
| 修炼受阻 | 法阵破碎、红色灵纹闪烁 |
| 修炼完成 | 金光落下、经验灵石飞入 |
| 进化 | 云气包裹、麒麟法相、破光化形 |

正式素材需适配 128×160 屏幕、RGB565、有限帧数和局部刷新，不能引入全屏持续闪烁。

## 8. 测试与验收

- 历练可持续提供经验，并能正常达到 LV12。
- 三个区域的历练经验分别为 1、2、3。
- 首领奖励按胜利次数持续减半且最低为 1。
- 旧存档可迁移，新字段不破坏现有进度。
- 成功 AI 任务按耗时奖励；失败、中间状态和重复任务不奖励。
- 三种来源均能被协议解析。
- 多 Hook 快速连续触发时串口发送不会互相抢占。
- AI 状态页可自动返回首页。
- 进化后立即保存并有清晰提示。
- 全部主机模型测试通过，Arduino 固件编译通过。
